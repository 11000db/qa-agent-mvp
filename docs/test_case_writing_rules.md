# Test Case Writing Rules

## 목적
이 문서는 QA Agent MVP에서 테스트 케이스를 일관된 방식으로 작성하기 위한 공통 규칙을 정의한다.
로그인, 회원가입 등 여러 기능에 대해 동일한 기준으로 테스트 케이스를 작성하고 리뷰할 수 있도록 한다.

## 공통 작성 규칙

### 1. 모든 테스트 케이스는 feature_name, priority, risk_level을 반드시 포함한다.
- 어떤 기능을 검증하는지 명확해야 한다.
- 테스트 우선순위와 리스크 수준이 정의되어야 한다.
- 이를 통해 어떤 테스트를 먼저 보고, 어디에 더 많은 검증을 할지 판단할 수 있어야 한다.

### 2. 모든 테스트 케이스는 happy path와 negative case를 함께 포함한다.
- 정상 동작만 확인하는 테스트 케이스는 충분하지 않다.
- 실패 조건, 예외 상황, 입력값 오류까지 함께 고려해야 한다.
- 실제 서비스 품질은 예외 상황에서 더 잘 드러난다.

### 3. expected_results는 추상 표현이 아니라 검증 가능한 문장으로 작성한다.
- "정상 동작한다", "잘 처리된다" 같은 표현은 사용하지 않는다.
- 화면 이동, 메시지 노출, 상태 변화처럼 실제로 확인 가능한 결과로 적는다.

좋은 예:
- 로그인 성공 후 메인 페이지로 이동해야 한다.
- 중복 이메일 입력 시 오류 메시지가 표시되어야 한다.

나쁜 예:
- 정상적으로 처리된다.
- 잘 동작한다.

### 4. preconditions와 test_data는 실제 실행 가능한 수준으로 구체적으로 작성한다.
- 테스트를 수행하기 전에 어떤 상태가 필요한지 분명해야 한다.
- 어떤 계정, 어떤 입력값, 어떤 환경이 필요한지 알 수 있어야 한다.
- 다른 사람도 같은 조건으로 테스트를 재현할 수 있어야 한다.

### 5. 자동화 가치가 높은 기능은 automation_candidate를 true로 표시한다.
- 반복 수행 가능성이 높고 회귀 테스트 가치가 큰 기능은 자동화 대상으로 본다.
- 로그인, 회원가입, 장바구니, 결제 등 핵심 기능은 우선적으로 자동화 후보로 판단한다.

## 적용 대상
이 규칙은 아래와 같은 기능 테스트 케이스 작성에 공통 적용한다.

- 로그인
- 회원가입
- 장바구니
- 주문/결제
- 프로필 수정
- 비밀번호 재설정
- 기타 핵심 사용자 플로우

## 작성 원칙 요약
좋은 테스트 케이스는 단순한 절차 문서가 아니라,
요구사항을 검증 가능한 결과와 우선순위로 구조화한 QA 자산이어야 한다.
