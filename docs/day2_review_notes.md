# Day 2 Review Notes

## 리뷰 대상
- sample_outputs/login_test_case.json

## 잘된 점
- 로그인 기능의 happy path가 포함되어 있다.
- 잘못된 비밀번호, 존재하지 않는 이메일, 필수값 누락 등 negative case가 포함되어 있다.
- expected result가 비교적 구체적으로 작성되어 있다.
- 로그인 기능을 핵심 기능으로 보고 priority와 risk_level을 높게 둔 점이 적절하다.

## 보완이 필요한 점
- 로그인 성공 상태를 확인할 수 있는 UI 요소가 조금 더 구체적이면 좋겠다.
- 오류 메시지 문구가 실제 서비스 기준으로 더 명확히 정의되면 좋겠다.
- edge case 중 어떤 항목이 실제 우선 검증 대상인지 더 좁혀도 좋겠다.

## 추가하고 싶은 테스트 포인트
- 로그인 버튼 연속 클릭 시 중복 요청이 발생하지 않는지 확인
- 엔터 키 입력으로 로그인 가능한지 확인
- 로그인 실패 후 입력값 유지 여부 확인
- 비밀번호 마스킹 토글이 있다면 정상 동작 여부 확인

## 자동화 관점 메모
- 로그인은 회귀 테스트 가치가 높아서 자동화 우선순위가 높다.
- test_steps와 expected_results가 비교적 명확해서 Playwright 테스트로 변환하기 좋다.
- negative case도 자동화 대상으로 적합하다.

## 오늘 느낀 점
- 테스트 케이스는 단순히 절차를 적는 것이 아니라,
  어떤 요구사항을 어떻게 검증할지 구조적으로 표현하는 문서라는 점을 다시 확인했다.
- 좋은 테스트 케이스는 happy path만이 아니라 negative case와 expected result 명확성이 중요하다.
