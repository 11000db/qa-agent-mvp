# TC Quality Rubric

## 목적
이 문서는 생성된 테스트 케이스가 QA 관점에서 충분히 쓸만한지 평가하기 위한 기준이다.
단순히 문장이 자연스러운지보다, 실제 테스트 설계에 필요한 품질 요소가 포함되어 있는지를 본다.

---

## 평가 기준

### 1. 요구사항 반영 여부
- 테스트 케이스가 실제 요구사항을 반영하는가?
- 핵심 기능이 빠지지 않았는가?
- 요구사항과 테스트 항목 간 연결(traceability)이 가능한가?

점검 질문:
- 이 테스트는 어떤 요구사항을 검증하는가?
- 핵심 성공 조건이 테스트에 포함되었는가?

---

### 2. Happy Path 포함 여부
- 정상 흐름이 빠지지 않았는가?
- 대표 사용자 시나리오가 포함되었는가?

점검 질문:
- 사용자가 가장 일반적으로 수행하는 주요 흐름이 포함되었는가?
- 성공 시 결과가 검증 가능하게 적혀 있는가?

---

### 3. Negative Case 포함 여부
- 실패/예외 시나리오가 포함되었는가?
- 잘못된 입력, 권한 없음, 중복 데이터 등 실패 조건을 고려했는가?

점검 질문:
- 잘못된 입력에 대한 테스트가 있는가?
- 실패 시 시스템이 어떻게 반응해야 하는지 정의되어 있는가?

---

### 4. Edge Case 포함 여부
- 경계값, 빈 값, 최소/최대값, 특수 입력 등을 고려했는가?

점검 질문:
- 경계값 테스트가 있는가?
- 일반 시나리오 외의 예외적인 조건도 포함되었는가?

---

### 5. Expected Result 명확성
- 예상 결과가 모호하지 않은가?
- “정상 동작한다” 같은 표현 대신 구체적으로 적혀 있는가?

좋은 예:
- 오류 메시지 "비밀번호가 일치하지 않습니다"가 표시된다.
- 로그인 성공 후 메인 페이지로 이동한다.

나쁜 예:
- 정상적으로 처리된다.
- 잘 동작한다.

---

### 6. Preconditions 명확성
- 테스트 전에 필요한 상태가 정의되어 있는가?
- 로그인 필요 여부, 테스트 데이터 존재 여부 등이 적혀 있는가?

점검 질문:
- 이 테스트를 실행하기 전에 무엇이 준비되어야 하는가?
- 환경/계정/데이터 조건이 분명한가?

---

### 7. 재현 가능성
- 다른 사람이 봐도 동일하게 실행 가능한가?
- 테스트 절차가 구체적인가?

점검 질문:
- 단계가 너무 추상적이지 않은가?
- 누가 봐도 따라할 수 있는가?

---

### 8. 우선순위와 리스크 판단
- 모든 테스트가 동일하게 다뤄지지 않고 중요도가 구분되어 있는가?
- 비즈니스 영향도와 장애 가능성이 반영되어 있는가?

점검 질문:
- 이 테스트가 왜 P0/P1/P2인지 설명 가능한가?
- 장애 시 영향이 큰 기능이 우선 반영되었는가?

---

### 9. 자동화 가능성
- 자동화에 적합한 테스트인지 판단 가능한가?
- 반복 실행 가치가 있는가?

점검 질문:
- 이 테스트는 Playwright/API 자동화 후보인가?
- 회귀 테스트에 적합한가?

---

### 10. QA 관점의 실용성
- 실제 현업에서 바로 수정/활용 가능한 수준인가?
- 너무 장황하거나 너무 비어 있지 않은가?

점검 질문:
- 이 테스트 케이스를 리뷰 후 바로 사용할 수 있는가?
- QA 엔지니어가 추가 보완할 양이 과도하지 않은가?

---

## 간단 점수 기준 예시

각 항목을 아래 기준으로 평가할 수 있다.

- 0점: 없음 / 거의 쓸 수 없음
- 1점: 일부 반영 / 보완 필요
- 2점: 충분히 반영 / 실무 사용 가능

총점 예시:
- 16점 이상: 우수
- 12~15점: 사용 가능하나 보완 필요
- 11점 이하: 재작성 권장

---

## 리뷰 메모 예시

- happy path는 있으나 negative case가 부족함
- expected result가 추상적이라 검증 포인트 보완 필요
- precondition과 test data가 부족함
- 우선순위 판단은 적절함
